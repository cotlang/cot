# DocC-Style Documentation System for Cot

**Date:** 2026-03-29
**Status:** Design document
**Goal:** Replace fragile line-number ac/ docs with symbol-linked documentation that powers IDE hover, cot.ac website, and educational content

---

## The Problem

The current ac/ documentation references source code by line number (`## Lines 379-503: parseFnDeclFull`). Every edit shifts line numbers, making the docs stale within hours. This is unsustainable.

## The Solution: Symbol-Based Documentation

Adopt Apple's DocC architecture adapted for Cot:

1. **Source code stays clean** — only `///` summary comments on declarations
2. **Rich docs live in ac/** — articles, tutorials, diagrams reference code by **symbol path**
3. **A Symbol Graph bridges them** — the compiler emits a JSON index of all symbols
4. **The LSP serves docs** — hover popups show rich content from ac/
5. **cot.ac renders docs** — the educational website is generated from the same source

---

## Architecture

```
Cot Source (.zig/.cot)
    │
    ├──► Cot Compiler (--emit-symbol-graph)
    │        │
    │        ▼
    │    symbol-graph.json
    │        │
    ▼        ▼
/// comments  +  ac/ catalog  ──► cot-doc compiler ──► .cotdoc archive
                                                            │
                                                  ┌────────┴────────┐
                                                  ▼                 ▼
                                            LSP hover          cot.ac website
```

### Component 1: Symbol Graph

The Cot compiler emits a JSON file listing every public declaration:

```json
{
  "module": "libcot-zig",
  "symbols": [
    {
      "id": "Parser",
      "kind": "struct",
      "path": ["libcot-zig", "parser", "Parser"],
      "location": { "file": "parser.zig", "line": 33 },
      "docComment": "Recursive descent parser producing a compact, data-oriented AST."
    },
    {
      "id": "Parser.parseFnDeclFull",
      "kind": "method",
      "path": ["libcot-zig", "parser", "Parser", "parseFnDeclFull"],
      "parent": "Parser",
      "location": { "file": "parser.zig", "line": 397 },
      "docComment": null
    }
  ],
  "relationships": [
    { "kind": "memberOf", "source": "Parser.parseFnDeclFull", "target": "Parser" }
  ]
}
```

For Phase 1, this can be generated by a simple Zig tool that parses `///` comments and `pub` declarations — it doesn't need full semantic analysis. Later, the checker can emit a richer graph.

### Component 2: ac/ Catalog (Restructured)

Current structure (line-number based):
```
ac/src/libcot-zig/
├── parser.md         ← one big file, line numbers everywhere
├── ast.md
└── types.md
```

New structure (symbol-path based):
```
ac/src/libcot-zig/
├── parser.docc/
│   ├── parser.md              ← module overview (no line numbers)
│   ├── Parser.md              ← ``Parser`` struct documentation
│   ├── Parser.parseFnDeclFull.md  ← deep dive on one function
│   ├── Parser.parseExpr.md
│   ├── tutorials/
│   │   └── WritingAParser.tutorial  ← step-by-step tutorial
│   └── resources/
│       └── ast-data-layout.png
├── ast.docc/
│   ├── ast.md
│   ├── Node.md
│   ├── Tag.md
│   ├── full.md
│   └── BuiltinKind.md
└── types.docc/
    ├── types.md
    ├── TypeRegistry.md
    ├── BasicKind.md
    └── ValueWitnessTable.md
```

Each `.md` file's heading declares which symbol it documents:

```markdown
# ``Parser.parseFnDeclFull``

Parses a complete function declaration including type params, value
params, return type, where clause, and body.

## Overview

This function handles all function modifiers through the ``FnModifiers``
struct. The parsing follows Cot's two-paren convention for generic functions:
`fn max(T)(a: T, b: T) T` — bare identifiers without `:` are type params.

## Topics

### Related
- ``Parser.parseFieldList``
- ``Parser.parseWhereClause``
- ``FnModifiers``
- ``ast.FnDecl``
```

Symbol references use double backticks: `` ``Parser.parseFieldList`` ``. These resolve via the symbol graph — no line numbers.

### Component 3: cot-doc Compiler

A build tool that:
1. Reads the symbol graph JSON
2. Reads the ac/ catalog markdown files
3. Merges them: matches `# ``SymbolPath`` ` headings to symbol graph entries
4. Validates all symbol links (warns on broken references)
5. Outputs:
   - JSON archive (for LSP and website)
   - Static HTML (for cot.ac)

For Phase 1, this can be a simple Zig program. Later, it could use DocC's open-source renderer (Vue.js) for the website.

### Component 4: LSP Integration

The rewritten LSP adds a documentation provider:

```
textDocument/hover request for "parseFnDeclFull"
    │
    ▼
LSP looks up symbol in symbol graph → "Parser.parseFnDeclFull"
    │
    ▼
LSP finds ac/src/libcot-zig/parser.docc/Parser.parseFnDeclFull.md
    │
    ▼
Returns markdown content as hover response
```

The LSP already handles hover via `textDocument/hover`. Adding documentation lookup is a few hundred lines — check the symbol graph index, find the matching ac/ file, return its content as markdown.

### Component 5: cot.ac Website

The cot-doc compiler outputs static HTML that becomes the cot.ac educational website. Each symbol page shows:

- The `///` summary from source
- The rich documentation from ac/
- Syntax-highlighted source code
- Links to related symbols
- Tutorials and guides

This is what Apple's developer.apple.com looks like for Swift — browsable, searchable, cross-linked.

## Migration Plan: Current ac/ → DocC-style

### Phase 1: Restructure ac/ files (1-2 days)

Convert each `<module>.md` into a `<module>.docc/` folder:
- Split the single large file into per-symbol files
- Replace all line number references with symbol paths
- Keep the content — it's good, just needs relinking

Example: `ac/src/libcot-zig/parser.md` (600 lines) becomes:
```
parser.docc/
├── parser.md          ← overview section from old file
├── Parser.md          ← struct docs
├── Parser.parseFnDeclFull.md   ← function docs
├── Parser.parseExpr.md
├── Parser.parseType.md
├── Parser.parseStmt.md
└── ...
```

### Phase 2: Symbol graph emission (1 day)

Write a simple tool that scans `.zig` files for `pub const/fn/var` declarations and `///` comments, emits `symbol-graph.json`. Doesn't need the full compiler — just declaration scanning.

### Phase 3: Validation tool (half day)

A tool that reads `symbol-graph.json` and all `ac/**/*.md` files, verifies every `` ``SymbolPath`` `` reference resolves, reports broken links. Run in CI.

### Phase 4: LSP documentation provider (1 day)

Add to the rewritten LSP: on hover, look up symbol in the ac/ catalog, return rich markdown. Falls back to `///` comment if no catalog entry exists.

### Phase 5: cot.ac static site generator (2-3 days)

Generate browsable HTML from the merged symbol graph + catalog. Can start simple (markdown → HTML) and later adopt DocC's Vue.js renderer for the polished look.

## Why Not Just Use DocC Directly?

DocC is Swift-specific:
- Requires the Swift compiler's `-emit-symbol-graph` flag
- Symbol Graph format assumes Swift's type system (protocols, extensions, generics)
- The `.tutorial` format is locked to DocC's renderer

But we can **adopt the architecture** (symbol graph + external catalog + merger tool) and **reuse the renderer** (swift-docc-render is a standalone Vue.js app that reads JSON). The JSON archive format is well-documented by SymbolKit.

The Cot-specific additions:
- Pipeline stage annotations (which compilation phase uses this symbol)
- ARC classification (trivial/non-trivial) in symbol metadata
- Concurrency annotations (Sendable, actor-isolated) in symbol metadata
- Educational content (how the compiler works, not just API reference)

---

## What Changes in Source Code

Almost nothing. The only addition to source files:

```zig
/// Recursive descent parser producing a compact, data-oriented AST.
pub const Parser = struct {
```

The `///` comment is the **summary** — one or two lines. All deep documentation lives in ac/. The symbol graph tool extracts the `///` and the declaration. The cot-doc compiler merges it with the ac/ catalog.

No `@doc:tag` annotations needed. No special comments. Symbol resolution uses the declaration name itself. This is exactly what DocC and Rust's rustdoc do.
